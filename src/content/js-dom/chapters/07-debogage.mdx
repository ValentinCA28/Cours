## Les méthodes de console

La <ic>console</ic> est ton premier outil de débogage. Tu la connais déjà avec `console.log()`, mais elle offre bien d'autres méthodes utiles.

### console.log, warn, error

```js
console.log("Information classique");        // Texte blanc/noir
console.warn("Attention, quelque chose ne va pas"); // ⚠️ Jaune
console.error("Erreur critique !");           // ❌ Rouge
```

Chaque méthode a un niveau de sévérité différent, ce qui te permet de **filtrer** les messages dans la console du navigateur.

### console.table

Affiche un tableau ou un objet sous forme de **tableau interactif** — beaucoup plus lisible qu'un `log` classique :

```js
const users = [
  { name: "Alice", age: 25 },
  { name: "Bob", age: 30 },
  { name: "Charlie", age: 22 },
];

console.table(users);
// Affiche un joli tableau avec colonnes name et age
```

### console.dir

Affiche les **propriétés** d'un objet de manière structurée. Très utile pour inspecter un élément DOM :

```js
const button = document.querySelector("button");
console.dir(button); // Montre toutes les propriétés de l'élément
```

### console.group / console.groupEnd

Permet de **grouper** plusieurs messages pour garder la console organisée :

```js
console.group("Chargement des données");
console.log("Requête envoyée...");
console.log("Réponse reçue : 200 OK");
console.log("3 utilisateurs chargés");
console.groupEnd();
```

### console.time / console.timeEnd

Mesure le **temps d'exécution** d'un bloc de code :

```js
console.time("rendu");

// Code dont tu veux mesurer la durée
for (let i = 0; i < 10000; i++) {
  document.createElement("div");
}

console.timeEnd("rendu");
// Affiche : rendu: 2.345ms
```

## Erreurs courantes

### TypeError : accès à une propriété de null/undefined

C'est l'erreur **la plus fréquente** en JavaScript DOM. Elle arrive quand tu essaies d'accéder à une propriété ou une méthode sur une valeur qui est <ic>null</ic> ou <ic>undefined</ic>.

```js
// ❌ Si l'élément n'existe pas, querySelector retourne null
const title = document.querySelector(".titre-inexistant");
console.log(title.textContent);
// TypeError: Cannot read properties of null (reading 'textContent')
```

**Causes fréquentes :**
- Faute de frappe dans le sélecteur CSS
- L'élément n'existe pas dans le HTML
- Le script s'exécute **avant** que le DOM soit prêt

**Solutions :**

```js
// Solution 1 : vérifier que l'élément existe
const title = document.querySelector(".titre");
if (title) {
  title.textContent = "Nouveau titre";
}

// Solution 2 : optional chaining (?.)
const text = document.querySelector(".titre")?.textContent;
// Si l'élément est null, text vaudra undefined au lieu de crasher
```

### Script exécuté avant le DOM

Si ton script se trouve dans le `<head>` sans l'attribut <ic>defer</ic>, il s'exécute **avant** que le HTML soit lu. Les éléments n'existent pas encore !

```html
<!-- ❌ Le script s'exécute avant que le body soit lu -->
<head>
  <script src="app.js"></script>
</head>

<!-- ✅ Solution 1 : attribut defer -->
<head>
  <script src="app.js" defer></script>
</head>

<!-- ✅ Solution 2 : script en bas du body -->
<body>
  <h1>Mon site</h1>
  <script src="app.js"></script>
</body>
```

## L'opérateur optional chaining (?.)

L'opérateur <ic>?.</ic> (optional chaining) est une manière élégante de **court-circuiter** l'accès à une propriété si la valeur est `null` ou `undefined`. Au lieu de crasher, l'expression retourne simplement `undefined`.

```js
// Sans optional chaining — risque de TypeError
const user = getUser();
const city = user.address.city; // 💥 crash si user ou address est null

// Avec optional chaining — sécurisé
const city = user?.address?.city; // undefined si user ou address est null
```

C'est particulièrement utile avec le DOM :

```js
// Modifier un élément seulement s'il existe
document.querySelector("#notification")?.classList.add("visible");

// Appeler une méthode seulement si l'élément existe
document.querySelector("form")?.addEventListener("submit", handleSubmit);
```

## Les breakpoints dans DevTools

Les <ic>console.log</ic> c'est bien, mais les **breakpoints** c'est beaucoup plus puissant. Un breakpoint **met en pause** l'exécution du code à un endroit précis, te permettant d'inspecter toutes les variables à ce moment-là.

<Callout variant="success" title="Workflow de débogage avec les DevTools">
  1. Ouvre les DevTools avec **F12** (ou Ctrl+Shift+I)
  2. Va dans l'onglet **Sources**
  3. Trouve ton fichier JavaScript dans l'arborescence à gauche
  4. **Clique sur un numéro de ligne** pour poser un breakpoint (un point bleu apparaît)
  5. Déclenche l'action qui exécute ce code (clic sur un bouton, etc.)
  6. L'exécution se met en **pause** à la ligne du breakpoint
  7. Inspecte les variables dans le panneau **Scope** à droite
  8. Utilise les boutons de contrôle pour avancer pas à pas :
     - **Step over** (F10) : exécute la ligne et passe à la suivante
     - **Step into** (F11) : entre dans la fonction appelée
     - **Resume** (F8) : reprend l'exécution normale
</Callout>

## Le mot-clé debugger

Tu peux aussi poser un breakpoint **directement dans ton code** avec le mot-clé <ic>debugger</ic>. Quand le navigateur rencontre cette instruction (et que les DevTools sont ouverts), il met l'exécution en pause comme un breakpoint classique.

```js
function calculateTotal(items) {
  let total = 0;

  items.forEach(function (item) {
    debugger; // L'exécution se met en pause ici
    total += item.price * item.quantity;
  });

  return total;
}
```

<Callout variant="warn" title="Pense à retirer debugger">
  Le mot-clé `debugger` est pratique pour le développement, mais **n'oublie pas de le retirer** avant de mettre ton code en production. Il ne fait rien si les DevTools sont fermés, mais c'est une mauvaise pratique de le laisser.
</Callout>

## Erreurs courantes avec fetch

Comme on l'a vu dans le chapitre précédent, <ic>fetch()</ic> a un comportement qui surprend souvent les débutants : il ne lance **pas** d'erreur pour les codes HTTP 4xx ou 5xx.

```js
// ❌ Ce code ne détecte PAS une erreur 404
async function loadData() {
  try {
    const response = await fetch("/api/data");
    const data = await response.json(); // Peut crasher si la réponse n'est pas du JSON
    return data;
  } catch (error) {
    console.error(error); // N'attrape que les erreurs réseau !
  }
}

// ✅ Version corrigée
async function loadData() {
  try {
    const response = await fetch("/api/data");

    if (!response.ok) {
      throw new Error(`Erreur HTTP ${response.status} : ${response.statusText}`);
    }

    const data = await response.json();
    return data;
  } catch (error) {
    console.error("Erreur lors du chargement :", error.message);
  }
}
```

## Résumé des outils de débogage

| Outil | Quand l'utiliser |
|---|---|
| <ic>console.log</ic> | Vérifier rapidement une valeur |
| <ic>console.table</ic> | Inspecter un tableau d'objets |
| <ic>console.dir</ic> | Voir les propriétés d'un élément DOM |
| <ic>console.time</ic> | Mesurer les performances |
| **Breakpoints** | Déboguer un flux complexe pas à pas |
| <ic>debugger</ic> | Poser un breakpoint temporaire depuis le code |
| <ic>?.</ic> | Éviter les crashes sur null/undefined |
